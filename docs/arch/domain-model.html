Environment 统一管理 Instance 的 spawn 决策。所有触发者（用户、IMChannel、autoStart）不直接 spawn，而是通过 Environment。Environment 的 workspacePath 列已废弃，实际路径由 workspace-resolver.ts 实时计算为 {WORKSPACE_ROOT}/{orgId}/{userId}/{envId}。

消息流 1：前端 ↔ Agent

用户在控制面板发消息。RCS 通过 relay 中继前端与 Agent（acp-link）之间的 WebSocket 通信。Session 元数据由 agent_session 表存储，对话细节由 acp-link 管理。

sequenceDiagram
    participant FE as 前端 React
    participant Env as Environment
    participant Inst as Instance
    participant Core as CoreRuntimeFacade

    FE->>Env: 请求连接 Agent
    Env->>Env: ensureRunning（检查 + spawn）
    alt 无 Instance
        Env->>Inst: spawn 新 Instance
        Inst->>Core: launchInstance
    end
    Env-->>FE: 返回 relay 连接信息
    FE->>Inst: WS relay 连接

    Note over FE,Core: 双向消息透传（Session 对话由 acp-link 管理）
    FE->>Inst: 用户消息
    Inst->>Core: 转发
    Core->>Inst: Agent 回复 (streaming)
    Inst->>FE: 转发

    Note over FE,Core: RCS 存储 agent_session 元数据（id / title / status / source）

消息流 2：IMChannel（聊天平台 ↔ Agent）

飞书/Telegram 消息通过 IMChannel 路由到 Agent，回复发回去。一个 IMChannel 可以有多个 imChannelRoute 规则（1:N）。

sequenceDiagram
    participant IM as 飞书/Telegram
    participant IMChan as IMChannel
    participant Env as Environment
    participant Inst as Instance
    participant EB as EventBus

    IM->>IMChan: 用户发消息
    IMChan->>IMChan: 路由匹配 → 找到 Environment

    IMChan->>Env: 请求确保 Instance 运行
    Env->>Env: 检查 / spawn Instance
    Env-->>IMChan: Instance ready

    IMChan->>Inst: 发送 prompt
    Inst->>EB: publish inbound

    Note over IMChan,EB: 订阅 Agent 回复
    IMChan->>EB: subscribe (agent_message_chunk)
    EB-->>IMChan: 积累 streaming 文本
    EB-->>IMChan: prompt_complete → flush

    IMChan->>IM: 发送回复到聊天平台

消息流 3：ScheduledTask（HTTP Cron 触发器）

Task 定时发送 HTTP 请求到指定 URL，最常见的目标是 Workflow。Task 不绑定 Environment 或 AgentConfig——它是一个纯粹的 HTTP cron 触发器，任务定义字段是 url/method/headers/body。

sequenceDiagram
    participant Cron as node-schedule
    participant Task as Task Service
    participant WF as Workflow
    participant Env as Environment
    participant Inst as Instance
    Cron->>Task: cron 触发
    Task->>Task: 检查防并发（runningTasks Set）
    Task->>WF: fetch(task.url, HTTP 请求)
    WF->>Env: 请求执行 Agent 步骤
    Env->>Env: ensureRunning
    Env->>Inst: 发送 prompt
    Inst-->>WF: Agent 回复
    WF-->>Task: 执行结果

消息流 4：Machine 远程节点部署

Instance 可指定部署到远程 Machine。Machine 通过 /acp/ws 注册并维持心跳，RCS 通过 agentMachineCache 路由消息到正确的远程节点。

sequenceDiagram
    participant Machine as Remote Machine
    participant WS as /acp/ws Handler
    participant Cache as agentMachineCache
    participant Env as Environment
    participant Inst as Instance

    Machine->>WS: 注册（含 machineId / capabilities）
    WS->>Cache: registerMachine
    Note over Machine,WS: 周期性心跳（heartbeat 检测超时）

    Env->>Env: spawnInstanceFromEnvironment
    Env->>Inst: launchInstance（指定 machineId）
    Inst->>Cache: 查 machineId → 路由消息
    Inst->>Machine: 通过 WS 发送指令

    Note over Machine,Inst: Machine 超时 → 清理旧 relay → 前端重连

消息流 5：用户注册 → 自动创建个人组织

新用户注册后，better-auth 的 databaseHooks.user.create.after 钩子自动创建 Organization 并授予 owner 角色。注册流程依赖 encryption.ts（AES-256-GCM 前端加密密码，服务端解密存入 account 表）。

sequenceDiagram
    participant FE as 前端
    participant Auth as /api/auth/*
    participant Hook as user.create.after 钩子
    participant DB as PostgreSQL

    FE->>Auth: POST sign-up（密码经 encryption.ts 加密）
    Auth->>DB: INSERT user
    DB-->>Auth: user.id
    Auth->>Hook: 触发 after hook
    Hook->>DB: INSERT organization（name=用户名, slug=personal-{id前8位}）
    Hook->>DB: INSERT member（userId, orgId, role=owner）
    Auth-->>FE: 成功 + session cookie

    Note over FE,DB: 后续：提取 x-active-org-id > query > cookie → AuthContext

权限与资源关系

不同资源在个人 (@user)、组织内 (@org)、登录用户 (@auth)、公开 Public 四个维度的可见性对照。
路由层通过 requireOrgScope(authContext, resourceOrgId) 统一拦截组织边界（src/plugins/require-team-scope.ts）。
