消息流 1：前端 ↔ Agent

用户在控制面板发消息。Session 由 acp-link 管理，RCS 只做消息透传。

sequenceDiagram
    participant FE as 前端 React
    participant Env as Environment
    participant Inst as Instance
    participant Agent as acp-link

    FE->>Env: 请求连接 Agent
    Env->>Env: 确保有 running Instance
    alt 无 Instance
        Env->>Inst: spawn 新 Instance
    end
    Env-->>FE: 返回 relay 连接信息
    FE->>Inst: WS relay 连接

    Note over FE,Agent: 双向消息透传（Session 由 acp-link 管理）
    FE->>Inst: 用户消息
    Inst->>Agent: 转发
    Agent->>Inst: Agent 回复 (streaming)
    Inst->>FE: 转发

    Note over FE,Agent: Session 列表通过 ACP 透传
    FE->>Agent: session/list (ACP 协议)
    Agent-->>FE: 返回 Session 列表

消息流 2：IMChannel（聊天平台 ↔ Agent）

飞书/Telegram 消息通过 IMChannel 路由到 Agent，回复发回去

sequenceDiagram
    participant IM as 飞书/Telegram
    participant IMChan as IMChannel
    participant Env as Environment
    participant Inst as Instance
    participant EB as EventBus

    IM->>IMChan: 用户发消息
    IMChan->>IMChan: 路由匹配 → 找到 Environment

    IMChan->>Env: 请求确保 Instance 运行
    Env->>Env: 检查 / spawn Instance
    Env-->>IMChan: Instance ready

    IMChan->>Inst: 发送 prompt
    Inst->>EB: publish inbound

    Note over IMChan,EB: 订阅 Agent 回复
    IMChan->>EB: subscribe (agent_message_chunk)
    EB-->>IMChan: 积累 streaming 文本
    EB-->>IMChan: prompt_complete → flush

    IMChan->>IM: 发送回复到聊天平台

消息流 3：ScheduledTask（HTTP Cron 触发器）

Task 定时调 URL，最常见的目标是 Workflow。Workflow 收到请求后通过 Environment 操作 Agent。

核心概念详解

👥 Team 团队（资源所有者）

资源所有权单位 | API Key 归 Team 所有 | 成员通过角色控制权限

Team 是 RCS 中所有资源的所有权单位。Environment、AgentConfig、Provider、Skill、McpServer、KnowledgeBase、IMChannel、ScheduledTask、API Key 都归 Team 所有。API Key 是 Team 级别的资源，不是 User 个人持有的——团队成员通过 Cookie Session 登录后切换到 Team，即可使用该 Team 的所有资源；外部系统通过 API Key 访问时，也是绑定到 Team。每个 Team 有三种角色：owner（管理团队）、admin（读写全部资源）、member（读全部，只写自己创建的）。用户注册后自动创建个人团队，也可以加入或创建其他团队。

🔵 Environment 资源管理层

核心枢纽 | 调度 Instance、统一 spawn 决策

Environment 是 RCS 最核心的概念，负责调度 Agent Instance 的生命周期（spawn / stop / autoStart）。它通过 agentConfigId（UUID）强绑定一个 AgentConfig，运行时将配置注入到 Agent workspace。所有触发者（用户手动、autoStart、IMChannel）不直接 spawn Instance，而是通过 Environment 统一决策。

🟢 Instance 运行实例

纯内存态 | 由 Environment 调度

一个 acp-link 子进程，有独立的端口和 PID。由 Environment 负责创建和管理生命周期。一个 Environment 可以有多个 Instance（并发处理多个用户/请求）。服务器重启后 Instance 消失，由 Environment 的 autoStart 机制恢复。

🟡 Session 会话（已下沉，非 RCS 领域概念）

由 acp-link 管理 | RCS 只做 ACP 消息透传

Session 是用户和 Agent 之间的一次对话记录，完全由 acp-link 进程管理。RCS 不存储 Session 元数据、不管理生命周期。前端通过 ACP 通道（session/list、session/load）直接与 Agent 交互，RCS 只做消息透传。

🔴 AgentConfig Agent 配置

配置中心 | 引用独立资源，被 Environment 强引用

Agent 行为的完整配置：用哪个 Model（引用 Provider 下的 Model）、系统提示词、权限规则。AgentConfig 不拥有任何资源，只是引用它们——Skill 是独立资源被 AgentConfig 引用，KnowledgeBase 通过 MCP 协议关联，McpServer 通过配置引用，Provider（含 Model）也是被引用的独立资源。Environment 通过 agentConfigId（UUID）强绑定 AgentConfig，改名不影响关联。内置 Agent（build/plan/general 等）不可删除。

⚪ Provider 服务商（含 Model）

独立资源 | Provider 包含 Model 列表 | 运行时注入

Provider 是 AI 服务商（OpenAI、Anthropic 等），包含该服务商下的多个 Model。由 RCS 集中管理在 PostgreSQL 中，运行时注入到 Agent workspace。API Key 不存明文，响应只返回 keyHint。AgentConfig 的 model 字段引用 Provider 下的某个 Model。Model 不作为独立领域概念，管理入口在 Provider 详情页内。

🟣 KnowledgeBase 知识库

独立资源 | 通过 MCP 协议与 AgentConfig 关联 | 多对多

用户上传的文档集合，由外部 Provider 做向量索引。通过 MCP 协议与 AgentConfig 关联——Agent 运行时通过 MCP 端点查询知识库。一个知识库可以给多个 Agent 用，一个 Agent 可以查多个知识库。AgentConfig 中配置 KB 的 MCP 端点，运行时注入。知识库是独立资源，不隶属于 AgentConfig。

🟠 Skill 技能

独立资源 | 被 AgentConfig 引用

Skill 是 Markdown 格式的指令文件，给 Agent 补充特定领域的知识和操作指南。Skill 是独立资源，被 AgentConfig 引用。元数据存在数据库，内容存在文件系统。支持全局和 workspace 两种 scope。

🟠 McpServer MCP 服务器

独立资源 | 被 AgentConfig 引用

MCP (Model Context Protocol) 服务器是给 Agent 提供外部工具的独立资源，被 AgentConfig 引用。支持 local（命令行启动）和 remote（URL 连接）两种类型。配置在 Agent 运行时通过 workspace 的 opencode.json 注入。

💬 IMChannel IM 通道

用户资源 | 选择连接方式 + 配置路由规则

用户界面上的一等资源，用于对接外部聊天平台（飞书、Telegram、Discord 等）。用户创建 IMChannel 时选择连接方式、填写凭证、配置路由规则（哪个聊天群 → 哪个 Environment）。运行时显示连接状态（已连接/未连接/错误）。用户不需要感知底层 Hermes 网关的存在。消息到达时，IMChannel 通过 Environment 确保 Instance 运行，然后路由消息。

🔷 ScheduledTask HTTP Cron 触发器

独立调度单元 | 定时调 URL | Workflow 的触发入口

ScheduledTask 是纯粹的 HTTP cron 触发器——定时调一个 URL。不绑定 Environment、不绑定 AgentConfig、不包含任务描述。最常见的场景是定时触发一个 Workflow URL，由 Workflow 编排后续的 Agent 执行流程。也可以直接调任意 HTTP 端点。每次执行产生一条 ExecutionLog。

🟣 Workflow 编排引擎

独立领域模块 | 通过 Environment 操作 Agent | 与 Cron 互动

Workflow 是 RCS 的独立领域模块，负责编排 Agent 的多步执行流程。它通过 Environment 操作 Agent（不直接接触 Instance 或 acp-link），封装了"选择 Environment → 选择 AgentConfig → 执行步骤 → 收集结果"的完整流程。Workflow 提供 URL 入口，ScheduledTask 通过 HTTP 调用 Workflow URL 实现定时触发。用户也可以手动触发或通过 API 触发。Workflow 归 Team 所有。
