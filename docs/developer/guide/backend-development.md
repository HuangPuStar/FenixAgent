# 后端开发规范

本文档面向 FenixAgent 后端开发，约束目录组织、分层职责、数据库设计、API 设计、扩展接入、注释和架构文档同步方式。未特别说明时，以本文件、`CLAUDE.md` 与 `CONTRIBUTING.md` 为准；若三者存在冲突，以离代码更近、约束更具体的规则为准。

## 1. 目录结构说明

后端主要目录集中在 `src/`、`packages/`、`drizzle/`、`scripts/`、`docs/`。

### 1.1 `src/` 后端主服务源码

- `src/index.ts`：服务启动入口，负责装配插件、路由、启动阶段任务，例如数据迁移。
- `src/openapi.ts`：统一维护 OpenAPI / Scalar 文档插件与全局 tag 定义，供入口按 `/api` 和 `/web` 两套文档分别挂载。
- `src/routes/`：HTTP / WebSocket 路由定义。
- `src/routes/web/`：控制台前端使用的内部业务 API。
- `src/routes/api/`：对外开放的稳定 API。
- `src/routes/acp/`：ACP WebSocket / relay 相关路由。
- `src/routes/mcp/`：MCP 协议相关入口。
- `src/services/`：业务服务层，承载业务规则、事务边界、跨表编排、对外部能力的调用。
- `src/services/config/`：配置类资源的领域服务，按资源类型内聚。
- `src/services/workflow/`：工作流相关服务逻辑。
- `src/services/data-migrates/`：数据迁移脚本实现，供启动时迁移器按顺序执行。
- `src/repositories/`：低层数据访问封装，只做持久化操作，不承载业务规则。
- `src/db/`：数据库连接、Schema 定义等。
- `src/schemas/`：请求参数、响应结构、配置 body 等校验 Schema。
- `src/plugins/`：Elysia 插件、中间件、鉴权、错误处理等横切逻辑。
- `src/auth/`：认证、鉴权、token、加密等能力。
- `src/transport/`：WebSocket、SSE、relay、事件总线等传输层能力。
- `src/types/`：跨模块共享的类型定义。
- `src/utils/`：纯工具函数。
- `src/errors/`、`src/errors.ts`：错误类型、错误码、错误映射。
- `src/test-utils/`：测试 stub、测试辅助方法。
- `src/__tests__/`：后端测试。

### 1.2 `packages/` 可插拔能力包

- `packages/*` 放独立 workspace 包。
- 适合放运行时插件、协议适配器、SDK、可替换执行引擎、独立封装的服务接入层。
- 原则上主服务只依赖抽象和包导出的稳定接口，不直接耦合包内部实现细节。

### 1.3 `drizzle/` 数据库迁移目录

- `drizzle/*.sql`：Drizzle 生成的 DDL 迁移文件。
- `drizzle/meta/`：snapshot 和 journal，属于迁移链的一部分，必须一起提交。
- `drizzle/README.md`：迁移冲突、合并、数据迁移边界说明。

### 1.4 `scripts/` 脚本目录

- 放构建、迁移、检查、运维辅助脚本。
- 生产迁移入口为 `scripts/migrate.ts`。

### 1.5 `docs/`

- `docs/arch/`：当前架构和主逻辑说明，是长期维护的架构文档。
- `docs/adr/`：关键架构决策记录。

## 2. 逻辑分层说明

后端分层遵循“路由薄、服务厚、仓储纯”的原则。

### 2.1 Route 层职责

- `routes` 只负责协议接入，不承载业务规则。
- 负责参数解析、Schema 校验、鉴权接入、调用 service、映射 HTTP 响应。
- `routes` 不允许直接访问 `repository/dao`、数据库连接、文件系统细节或第三方 SDK。
- `routes` 不允许跨多个 service 自己编排复杂业务；编排逻辑应下沉到 service。

### 2.2 Service 层职责

- `services` 是唯一允许组织业务流程的层。
- `services` 可以调用 repository、其他 service、packages 导出的接口、文件系统和外部服务客户端。
- `services` 不应直接操作数据库连接、ORM 实例或 SQL 构造器；凡是数据库读写、查询条件拼装、事务内持久化步骤，都应先收敛到 `src/repositories/` 暴露的接口，再由 service 编排调用。
- 一个完整业务动作的事务边界、幂等控制、权限判断、跨表写入、日志记录，应优先收敛在 service。
- 同一张表或同一类资源的业务操作，尽量内聚在同一个 service 模块或同一组紧邻的 service 文件中，不要分散到很多“数据类”“工具类”里。

#### 2.2.1 Service 直接使用数据库的例外

- 默认情况下，业务 `service` 必须通过 repository 访问数据库，不把 `src/db/` 直接暴露为常规依赖。
- 若需要跨多个 repository 组织同一事务，优先让 repository 支持接收统一的 `db/tx` 上下文；service 负责开启事务和编排，不负责下沉为具体 SQL。
- 数据迁移、运维修复、一次性补算脚本等非长期业务 service，可基于实际需要直接使用 `db`，但应尽量限制影响范围，不把这类写法扩散回常规业务模块。
- 极少数无法合理沉淀到 repository 的复杂只读查询，可在评审后作为例外处理；但需要在实现处注明原因，避免后续演变成随意绕过 repository 的先例。

### 2.3 Repository / DAO 层职责

- `src/repositories/` 只做数据访问封装。
- repository 负责查询、插入、更新、删除、分页、条件拼装等持久化细节。
- repository 不承载业务判断，不做跨领域编排，不返回带业务副作用的结果。
- repository 默认不直接暴露给 route；只能由 service 调用。

### 2.4 推荐依赖方向

推荐依赖方向为：

`routes -> services -> repositories -> db`

`services -> packages/*`

`routes -> schemas`

禁止反向依赖，例如 repository 调用 route、service 反向依赖某个具体 route 文件。

### 2.5 表操作内聚规则

- 同一个表或同一类资源的主要数据库访问逻辑应集中在对应 repository 中维护。
- 确需拆分时，应按领域边界拆，而不是按 SQL 动作拆。

## 3. 数据库设计说明

数据库以 PostgreSQL + Drizzle ORM 为标准方案，`src/db/schema.ts` 是唯一 Schema 真相来源。

### 3.1 表设计基本原则

- 新增业务表必须有稳定主标识字段。
- 默认使用 `id` 作为主键字段名，类型可根据需求、可读性和可追踪性选择 `uuid` 或字符串 ID。
- 若受第三方框架约束或历史设计影响需要沿用既定字段定义，应保持与所在模块现状一致，不为了形式统一做高风险重构。
- 表之间关联统一使用稳定主标识关联，关联字段命名默认与被关联表主键保持一致。
- 禁止依赖名称、路径、可变 code、展示名等非稳定字段做外键语义关联。
- 所有核心业务表都应包含必要的时间字段，至少考虑 `createdAt`，可更新实体通常同时包含 `updatedAt`。
- 涉及组织隔离的资源，必须明确 `organizationId` 或等价租户字段。

### 3.2 关联关系设计

- 一对一：仅在强约束且生命周期一致时直接放外键。
- 一对多：若“多”的一侧有独立生命周期，可直接在子表放外键。
- 多对多：必须使用独立关联表，不在单字段中存数组关系。
- 即使当前看起来是一对多，只要未来可能扩展属性、排序、状态、来源，优先设计为独立关联表。
- 关联表命名应清晰表达关系，例如 `agentConfigSkill`、`agentKnowledgeBinding`。

### 3.3 字段设计要求

- 枚举语义优先用 `pgEnum` 或受限字符串，不允许魔法值散落在业务代码里。
- JSON 字段只用于确实不适合完全结构化、且读写边界清晰的数据。
- 可空字段必须有明确业务含义，不要为了“以后可能用到”随意 nullable。
- 允许空字符串和允许 `null` 是两个不同语义，默认值处理必须使用符合语义的方式，避免 `||` 吃掉合法空字符串。

### 3.4 数据库相关文件放置位置

- 表结构定义：`src/db/schema.ts`
- 数据库连接与导出：`src/db/`
- DDL 迁移文件：`drizzle/`
- 迁移执行入口：`scripts/migrate.ts`
- 数据迁移实现：`src/services/data-migrate.ts` 与 `src/services/data-migrates/`

### 3.5 Schema 变更流程

表结构变更按下面流程执行：

1. 修改 `src/db/schema.ts`
2. 执行 `bun run db:generate --name <migration-name>` 生成迁移
3. 执行 `bun run db:migrate` 应用迁移并在本地开发环境验证
4. 检查生成的 `drizzle/*.sql` 与 `drizzle/meta/*`
5. 补齐或更新相关 service、schema、测试、文档
6. 提交时必须连同整个 `drizzle/` 目录一起提交

### 3.6 生产迁移要求

- 开发环境和生产环境都应通过 `db:migrate` 这类基于迁移文件的方式执行变更，避免形成直接推平 schema 的习惯。
- 生产迁移统一通过 `scripts/migrate.ts` 构建出的 `migrate.js` 执行。
- 禁止手写 SQL 迁移文件绕过 Drizzle 生成流程。

### 3.7 数据迁移要求

DDL 迁移和数据迁移必须分离：

- 表结构变更走 `drizzle/`。
- 已有数据的批量搬迁、修复、补算、回填，走 `src/services/data-migrate.ts`。

数据迁移规则：

- 每个数据迁移在 `src/services/data-migrates/` 下独立成文件。
- 必须实现统一接口并注册到 `src/services/data-migrate.ts`。
- `name` 必须唯一，执行记录写入 `data_migrate_record`。
- 迁移逻辑必须幂等；失败后再次启动时应可安全重试。
- 不要把批量 `UPDATE/INSERT/DELETE` 数据修复逻辑直接写进 DDL SQL。

### 3.8 迁移冲突与故障处理

- 一个功能分支内若生成多个迁移节点，合并前应压缩为一个新增节点。
- 若本地迁移链与远端冲突，先同步远端迁移，再基于最新 schema 重新生成当前功能迁移。
- 不要手工拼接多个迁移 SQL 来“凑过冲突”。
- 若迁移执行失败，先确认数据库当前落点、`drizzle/meta/_journal.json`、本地 schema 是否一致，再决定回滚或重生迁移。
- 涉及生产迁移问题时，优先先定位当前数据库结构、迁移记录和代码 schema 的差异，再根据情况通过修正迁移文件重新执行、修正迁移记录、人工对齐等方式恢复一致性。
- 迁移冲突、压缩与数据迁移边界的更细规则，以 `drizzle/README.md` 为准。

## 4. API 设计说明

本项目维护两套 API：`/web` 和 `/api`。两者用途、调用方、兼容性要求不同，不能混用。

新增接口、修改接口或重写 route 时，先满足下面这些 API 设计约束，再补文档元数据。

### 4.0 基本设计规则

- API 功能必须单一、明确。正常情况下不要在一个接口里通过 `action` 等字段分支处理不同业务行为；只有明确要求，或 WebSocket、长连接事件流这类天然多消息类型场景，才允许这样设计。
- 设计、命名、鉴权和兼容性判断前，必须先明确接口属于 `/web/*` 还是 `/api/*`。

### 4.1 `/web` API

- 面向控制台前端和项目内部业务页面。
- 允许更贴近页面交互和控制面板场景。
- 统一放在 `src/routes/web/`。
- 默认返回 `{ success: true, data }` 或 `{ success: false, error }`。
- 历史上存在配置类 `action` 分发接口，新增接口时不默认沿用；若不是配置聚合型场景，应优先设计为单一职责接口。

### 4.2 `/api` API

- 面向外部系统、OpenAPI、API Key 调用方。
- 统一放在 `src/routes/api/`。
- 必须优先保证协议稳定和向后兼容。
- 返回结构应清晰、稳定、适合第三方接入，不随前端页面实现细节摆动。

### 4.2.1 OpenAPI 文档组织

- OpenAPI / Scalar 的统一装配入口放在 `src/openapi.ts`，不要把全局 tags、文档路径和插件配置散落到各 route 文件或 `src/index.ts` 中。
- 当前维护两套文档：
  - 对外 API 文档：面向 `/api/*`
  - 控制台 Web API 文档：面向 `/web/*` 和平台内部接口
- route 负责声明就近的 `detail`、schema 和响应结构；全局文档分组、tag 描述和文档挂载路径由 `src/openapi.ts` 统一维护。

### 4.3 路由设计规范

- URL 使用小写 kebab-case。
- 资源名优先使用复数，如 `/api/agents`、`/web/knowledge-bases`。
- URL 表达资源，动作优先由 HTTP 方法表达。
- 确实不是 CRUD 的行为，可使用动作后缀，如 `POST /api/sessions/:id/cancel`。
- 路径参数只放资源标识和层级关系，例如 `:id`、`:sessionId`；筛选、分页、排序、开关类参数放 query。
- 分页参数统一优先使用 `page`、`pageSize`；排序参数统一优先使用 `sortBy`、`sortOrder`；布尔筛选参数使用语义化命名，例如 `includeDisabled`、`withMembers`。
- `GET` 只做查询，不带 body。
- `POST` 用于创建或触发动作，更新优先 `PUT`，删除用 `DELETE`。

### 4.4 Route 编写规范

- 每个 route 文件应同时包含：
  - 参数/请求体/响应 Schema 绑定
  - 鉴权声明
  - OpenAPI 元数据
  - 调用对应 service 的实现
- route 内不要直接写复杂 SQL、跨表事务、批量状态迁移、文件系统落盘逻辑。
- route 的错误映射要稳定，对外错误码要可预期。
- `/web/*` 默认返回 `{ success: true, data }` 或 `{ success: false, error }`；没有业务数据时也优先返回 `{ success: true, data: ... }`，避免同一类接口有时返回 `data`、有时完全不返回。
- `/api/*` 对外接口不跟随 `/web/*` 使用 envelope；保持现有“成功返回裸对象或列表对象、失败返回 `{ error }`”的外部契约，避免内部控制台规范误伤对外兼容性。
- 错误响应统一返回 `error` 对象，至少包含 `code` 和 `message`；需要补充上下文时再增加字段，不要直接返回裸字符串或格式不固定的对象。
- `/api/*` 返回结构要稳定、可预测；列表接口优先返回对象结构而不是裸数组，至少区分列表数据和分页元信息，例如 `{ items, total, page, pageSize }`。

#### 4.4.1 Elysia + OpenAPI 标准写法

- route 上应补充 `detail`，并显式声明 `params`、`query`、`headers`、`body`、`response`。
- `summary`、`description`、tag 描述统一优先使用中文。
- schema 必须定义在 `src/schemas/` 目录中，禁止在 route 文件中内联声明请求体、响应体或字段结构。
- route 上声明 `params`、`query`、`headers`、`body`、`response` 时，默认直接使用 `src/schemas/` 中导出的 schema 实体；不要补 `any`。
- 字符串 model 引用仅适用于历史兼容或少量共享注册场景；新接口默认优先使用 schema 实体。
- 当 route 已声明 `response`，且实现中存在非 2xx 返回分支时，默认使用 `status(code, body)` 返回；不要优先使用 `error(code, body)`。
- 使用 `status(code, body)` 时，不要改变已有响应结构；除非明确要求更新结构体，否则如果前端已经依赖历史错误结构，必须保持原 body 兼容。
- 为 OpenAPI 展示补充必要的 `model` 注册，并为所属全局 tag 补充中文 `description`；全局 tag 定义统一维护在 `src/openapi.ts` 中。
- 如果接口属于内部使用、框架透传、静态资源、代理入口、MCP 服务入口、WebSocket、协议入口等不面向外部开发者的能力，也要补说明；需要隐藏时使用 `detail.hide: true`。

### 4.5 接口变更要求

- 默认保持向后兼容。
- 新增字段优先采用“追加不破坏”的方式。
- 删除字段、修改字段语义、修改默认值、改变错误结构，都视为破坏性变更。
- 无法兼容时，应新增新接口或新版本，而不是直接改坏旧接口。
- `/web` 即使属于内部 API，也要考虑现有前端和已有页面调用，避免静默破坏。
- 涉及接口变更时，必须同步更新：
  - route schema
  - 调用方
  - 测试
  - 相关开发文档或 OpenAPI 说明

## 5. 新类型工具 / 服务接入规范

新增工具、执行引擎、第三方服务、协议适配器时，统一按“抽象接口 + 独立 package + 主服务通过配置装配”的方式接入。

### 5.1 接入原则

- 可插拔
- 可替换
- 独立演进
- 低耦合
- 配置驱动

### 5.2 标准接入方式

1. 先定义该类型能力的抽象接口或稳定契约。
2. `src/services/` 中只面向抽象接口编程，不耦合某个具体实现。
3. 具体实现封装到 `packages/<name>/` 中。
4. 主服务通过配置、注册表或工厂选择实际启用的 package。
5. 若需要暴露页面或控制台能力，优先通过 route 转发或代理接入，不把 package 内部实现直接散落到主服务页面逻辑中。

### 5.3 代码组织建议

- 抽象定义放在主服务共享边界或稳定 package 中。
- 具体服务接入包内负责：
  - 第三方 SDK 封装
  - 配置适配
  - 错误转换
  - 协议适配
  - 独立测试
- 主服务只负责：
  - 配置读取
  - 生命周期管理
  - 领域编排
  - 权限与租户上下文控制

### 5.4 禁止事项

- 不要在 `src/services/` 里直接散落某个第三方服务的大量 SDK 调用。
- 不要让业务逻辑到处 `if provider === "xxx"` 分支。
- 不要把页面、路由、服务、数据结构全部绑死在一个具体外部实现上。

## 6. 注释说明

注释目标是帮助后来者理解设计原因和边界，不是重复代码表面行为。

### 6.1 必须写注释的地方

- 类、公共函数、导出工具、类型定义：必须写清晰简洁的文档注释。
- 对职责不直观、边界条件较多、被多处复用或名称不足以自解释的内部辅助函数，也应补充必要的文档注释。
- 非直观业务规则、兼容性处理、临时方案：写原因注释。
- 长函数中的关键处理阶段：必须用简短注释分段，帮助后续维护者快速理解结构。
- 容易误用的边界条件、幂等要求、顺序要求：必须注明。
- 数据迁移、协议兼容、权限判断、降级分支：应说明触发原因和风险。

### 6.2 注释写法要求

- 优先解释“为什么这样做”。
- 少解释“这行代码做了什么”。
- 临时方案要写清适用范围、清理条件和后续替换目标。
- 文档注释保持简洁，避免空话。

### 6.3 日志要求

- 涉及业务流程、状态变化、外部调用、异常处理的代码，默认应主动补齐必要日志，不要只在报错时随手打印一句。
- 关键输入输出、错误处理、外部调用失败、进入兜底逻辑、兼容性分支、重试、降级、跳过、提前返回和状态迁移时，应补充日志。
- 日志要优先记录关键上下文、分支原因、对象标识、结果状态和影响范围，避免只输出空泛文本。
- 临时兼容或降级路径的日志应明确说明触发原因和影响范围。
- `catch` 中必须保留错误日志，至少包含 `console.error(err)`，不要吞错。

## 7. 设计文档同步要求

主要逻辑的新增、重构、旧逻辑改造，都必须同步更新设计文档，保证文档描述的是当前真实架构，而不是历史状态。

### 7.1 需要同步文档的场景

- 新增核心模块
- 修改主链路
- 调整分层职责
- 变更领域模型、资源关系、权限边界
- 引入新的可插拔运行时、协议、外部服务
- 重构数据库模型、迁移方式、启动流程

### 7.2 文档放置建议

- 长期有效的架构说明放 `docs/arch/`
- 关键设计决策放 `docs/adr/`
- 功能级设计、计划、验收文档按当前仓库约定放在 `docs/` 下的对应专题目录中维护。

### 7.3 文档更新要求

- 代码合入前，相关架构文档必须同步更新。
- 文档要覆盖真实主流程、关键边界、依赖关系、限制条件。
- 如果实现偏离旧文档，应优先改文档，不允许放任文档长期失真。
- 影响多个模块时，至少检查 `docs/arch/` 下对应专题文档是否需要同步，如路由、数据库、配置、实例、权限、知识库、工作流等。

## 8. 开发落地清单

提交前至少自检以下事项：

- route 是否只调用 service，没有直接碰 repository / db
- service 是否通过 repository 访问数据库，没有直接操作 `src/db/` 或手写持久化细节
- service 是否收敛了该领域的核心业务逻辑
- 同一资源的表操作是否足够内聚
- schema 变更是否同时提交 `drizzle/` 完整产物
- 数据修复是否走了 `data-migrate`，而不是写进 DDL
- `/web` 和 `/api` 是否按用途分层，没有混淆
- 新接入的外部能力是否做到了抽象、可替换、低耦合
- 注释、日志、测试、架构文档是否同步更新
- 代码改动提交前是否通过 `bun run precheck`

## 9. 与现状兼容的补充说明

- 仓库中的业务表主键命名以 `id` 为主；涉及第三方框架约束或历史设计的表，继续按现状保留。
- 新规范优先约束“新增设计”和“增量重构”，不要求为了形式统一对稳定历史结构做高风险改表。
- 若要推动历史模块对齐本规范，应通过独立设计文档和迁移方案分步实施，避免一次性大改。
